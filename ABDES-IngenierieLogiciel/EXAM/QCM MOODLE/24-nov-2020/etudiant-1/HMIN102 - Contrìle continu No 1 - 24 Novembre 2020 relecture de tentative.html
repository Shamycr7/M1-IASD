public class Dice {

    private static final int nbFaces=6;
    private Random randomGen;

    public Dice(Random randomGen) {
        this.randomGen = randomGen;
    }

    public int roll() throws DiceException {
        int result = randomGen.nextInt(nbFaces) + 1; // 0 is not a possible value but nbFaces is.
        if (result > nbFaces || result<=0)
            throw new DiceException("Incorrect value");
        return result;
    }
}


On donne également la classe DiceBattle qui représente une bataille de deux dés, régie par les règles suivantes pour chaque round :
* à chaque round, les 2 dés sont lancés.
* si un des deux dés jette une DiceException, un message s'affiche et le round est annulé (rien ne se passe pour ce round)
* s'il y a égalité entre les valeurs des 2 dés, le round ne "compte pas", et le compteur de rounds n'est pas incrémenté et les scores des joueurs n'est pas modifié
* s'il n'y a pas égalité mais que la somme des valeurs des 2 dés fait 11, le compteur de rounds est incrémenté et chaque joueur marque 1 point.
* dans les autres cas, le joueur avec la plus forte valeur de dés gagne le round et marque 1 point, le perdant a son score inchangé. Le compteur de scores est incrémenté.

public class DiceBattle {
    private Dice dicePlayer1;
    private Dice dicePlayer2;

    private int pointsP1=0;
    private int pointsP2=0;

    private int nbRounds=0;

    public DiceBattle() {
        dicePlayer1=new Dice(new Random());
        dicePlayer2=new Dice(new Random());
    }

    public DiceBattle(Dice d1, Dice d2) {
        dicePlayer1=d1;
        dicePlayer2=d2;
    }

    public void round() {
        int vp1;
        int vp2;
        try {
            vp1=dicePlayer1.roll();
            vp2=dicePlayer2.roll();
            if (vp1!=vp2) {
                nbRounds++;
                if (vp1+vp2==11) {
                    pointsP1++;
                    pointsP2++;
                }else {
                    pointsP1+= (vp1>vp2)?1:0;
                    pointsP2+= (vp2>vp1)?1:0;
                }
            }
        } catch (DiceException e) {
            System.out.println("Something went wrong while rolling dices. The round is cancelled.");
        }
    }

    public void ThreeRoundsPlay() {
        while(nbRounds<3) {
            round();
        }
    }

    public int getPointsP1() {
        return pointsP1;
    }

    public int getPointsP2() {
        return pointsP2;
    }

    public int getNbRounds() {
        return nbRounds;
    }
}

On donne le code ci-dessous. L'ensemble de ces classes/interfaces vous parait-il mettre en œuvre le patron de conception Composite ?
Si oui vous expliciterez précisément les rôles joués par chacune des classes ou des instances de ces classes
Sinon vous expliquerez précisément en quoi l'ensemble de ces classes ne vous paraît pas mettre en oeuvre ce patron de conception.

interface A {
    public default void m1() {
        // a behavior
    }
    public void m2();
}
class B implements A{
    public void m2() {
        // a behavior
    }
}
class C implements A{
    protected ArrayList<A> components;

    public void addComponent(A a) {
        this.components.add(a);
    }
    
    public void m2() {
        components.forEach(c->c.m2());
    }
}
class D extends C {
    public void m2() {
        // a behavior
        super.m2();
    }
}

Ce code met en oeuvre le patron de conception composite, le role des classes est le suivant :
Classe A : est le composant, définit les opérations.
Classe B : est une feuille de notre patron.
Classe C : est un composé.
Classe D : est une spécialisation du composé mère. Il y a transtypage sur la méthode m2 qui se voit être celle de C, puis elle meme délègue la méthode m2 au composant de sont arraylist components lors de l'itération.

public class C extends A{

1) oui l'ensemble de ces classes/interface met en oeuvre la réalisation et l'utilisation d'un framework.


Elle est dotée d’un cœur (code) générique, extensible et adaptable. Le code du framework (pré-existant) ici A invoque (callback) les parties de code représentant la nouvelle application ici B ou C en un certain nombres d’endroits prédéfinis nommés (points d’extensions ou point de paramétrages ou encore «Hot spot») ici la méthode m1().

Notre injection de dépendance dans B et C ce fait sur le type de la méthode m2.
